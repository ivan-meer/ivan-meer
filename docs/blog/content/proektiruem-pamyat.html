Память — то, что отличает агента от чата. Чат отвечает на вопрос и забывает. Агент помнит проект, клиента, вчерашние решения — и не тащит весь архив в каждый запрос.

Когда я проектирую memory-слой для агентной системы, я раскладываю его на три уровня. Каждый решает свою задачу, и ни один не заменяет другой.

L1: Working Memory — краткосрочная

Это то, что лежит в окне контекста LLM. Самый простой слой, но именно здесь совершают больше всего ошибок.

Правило: контекст — это дефицитный ресурс. Каждый токен на истории диалога — это токен, не потраченный на инструменты, размышления и полезный контент.

# ❌ Так делать не надо — тащить всю историю:
messages = [
    {"role": "system", "content": SYSTEM_PROMPT},
    *all_history,  # 50 сообщений, 120K токенов
    {"role": "user", "content": new_query}
]

# ✅ Надо так — суммаризация и обрезка:
def build_context(history, working_mem_limit=32000):
    """Собирает контекст с суммаризацией старого."""
    recent = history[-6:]  # последние N сообщений
    older = history[:-6]

    if token_count(older) > working_mem_limit:
        summary = summarize_messages(older)
        return summary + recent
    return older + recent

Что реально работает в продакшене

  • Скользящее окно: 6–10 последних сообщений (2–4K токенов) + суммаризированная предыстория
  • Сжатие инструментов: после вызова инструмента храните не весь ответ, а только извлечённые факты
  • Тримминг по токенам: не по числу сообщений — разные сообщения весят по-разному
def trim_context(context, max_tokens=24000, reserve_for_output=4000):
    """Обрезает контекст, резервируя место под ответ."""
    budget = max_tokens - reserve_for_output
    while token_count(context) > budget and len(context) > 1:
        context.pop(1)  # удаляем самое старое, сохраняя system
    return context

L2: Long-term Memory — долгосрочная

Здесь начинается инженерия. Долгосрочная память — это RAG-система, встроенная в агента. Она отвечает на вопрос «что мой агент знал о проекте или клиенте раньше?».

Из чего состоит

  1. Чанкинг — нарезка документов на осмысленные куски
  2. Эмбеддинги — превращение текста в векторы
  3. Хранилище — векторная БД с гибридным поиском
  4. Реранжировка — отбор лучших результатов

Чанкинг: семантический > фиксированный

Фиксированная нарезка по N токенов — главная причина плохих результатов RAG.

# ❌ Фиксированные чанки
def fixed_chunks(text, size=256, overlap=20):
    words = text.split()
    return [' '.join(words[i:i+size]) for i
            in range(0, len(words), size - overlap)]

# ✅ Семантический чанкинг — по границам смысла
def semantic_chunks(text, max_size=512):
    """Разбивает по абзацам/заголовкам, не разрывая мысль."""
    paragraphs = re.split(r'\n\s*\n', text)
    chunks, current = [], []

    for p in paragraphs:
        if sum(len(c) for c in current + [p]) > max_size * 4:
            chunks.append('\n\n'.join(current))
            current = [p]
        else:
            current.append(p)

    if current:
        chunks.append('\n\n'.join(current))
    return chunks

Гибридный поиск: вектор + ключевые слова

Векторный поиск отлично находит «похожие по смыслу», но пасует на точных совпадениях (ID заказа, название компании). BM25 (ключевые слова) — наоборот. Вместе — идеальная пара.

-- Гибридный поиск в pgvector + pg_bm25 (ParadeDB)
SELECT
  doc.id,
  doc.content,
  (coalesce(1.0 / (row_number() OVER vector_rank + 60), 0.0) * 0.5 +
   coalesce(1.0 / (row_number() OVER keyword_rank + 60), 0.0) * 0.5) AS score
FROM documents doc
WHERE doc.embedding <=> $query_embedding < 0.8
   OR doc.content ILIKE '%' || $query_keyword || '%'
Продакшен-схема с Qdrant

В AdminHero ретривер работает так: запрос → query rewriting (уточнение через дешёвую LLM) → параллельный поиск (векторный + BM25) → RRF-слияние → реранжировка кросс-энкодером. Точность top-5 вырастает с ~60% до ~90%.

Реранжировка — обязательный шаг

Векторный поиск возвращает 10–20 чанков. Среди них — 3–4 релевантных, остальные — шум. Кросс-энкодер (Cohere rerank, BAAI/bge-reranker) пересчитывает релевантность каждой пары «запрос-чанк» и отсеивает мусор.

~90%

L3: Semantic Memory — граф знаний

Этот слой — вершина архитектуры. Вместо «поиска похожих документов» агент обращается к структурированному графу фактов.

# Извлечение фактов из диалога
facts = [
    {"subject": "client:acme-corp", "predicate": "uses", "object": "supabase"},
    {"project": "adminhero", "predicate": "deadline", "object": "Q3 2026"},
    {"subject": "client:acme-corp", "predicate": "contact", "object": "ivan@acme.com"},
]

# Запрос к графу
def query_semantic(agent, entity):
    return graph_store.query("""
        MATCH (n)-[r]->(m)
        WHERE n.id = $entity OR m.id = $entity
        RETURN n.id, type(r), m.id
    """, {"entity": entity})
Как это построено в AdminHero

После каждого шага агента запускается экстрактор фактов (LLM call на дешёвой модели). Факты сохраняются в Supabase с TTL. Перед ответом агент проверяет: «что я уже знаю об этой сущности?». Граф не обязан быть настоящей графовой БД — для 80% случаев хватает структуры {entity: [{fact, timestamp, source}]}.

Грамли, на которые я наступил

1. «Всё в контекст — и пусть модель сама разберётся»

Самая дорогая ошибка. Контекст 200K токенов — это не память, это свалка. Модель теряет фокус, а вы платите за каждый токен.

Решение: каждый слой памяти имеет бюджет токенов. Working Memory — 8K, RAG — 12K, Semantic — 2K. Жёстко.

2. Векторный поиск без ключевых слов

Вернёт чушь на точном запросе вроде «найди контракт #INV-2024-0891». Эмбеддинги не умеют в точные совпадения.

Решение: гибридный поиск. Всегда.

3. Один чанкер на все типы документов

Код и документацию нельзя чанкать одинаково. Код — по функциям и классам, документацию — по заголовкам, переписку — по дням.

Решение: мета-чанкер, который определяет тип контента и выбирает стратегию нарезки.

4. Отсутствие TTL и стратегий забывания

Память бесконечно растёт, агент тормозит, качество падает.

Решение: у каждого факта — TTL. Рабочая память чистится после каждой сессии. Долгосрочная — реиндексируется по расписанию (daily re-index, weekly re-chunk).

Выводы

  1. Память AI-агента — это система, а не поле в JSON. Проектируйте её как отдельный слой с собственной архитектурой.
  2. Три уровня — минимум для продакшена. Рабочая память для сессии, RAG для знаний, семантика для фактов.
  3. Гибридный поиск — не опция, а необходимость. Точность без ключевых слов проседает на 40% на предметных доменах.
  4. Реранжировка окупается с первой сотни запросов. Вы просто не видите, сколько мусора приходит из векторного поиска.
  5. Стратегия забывания важнее стратегии запоминания. Хранить всё легко. Решать, что забыть — инженерия.

Готовый сценарий: берёте pgvector (или Qdrant), добавляете семантический чанкинг, навешиваете гибридный поиск с RRF и кросс-энкодер сверху. Это — база. На этой базе можно строить агента, который помнит.
